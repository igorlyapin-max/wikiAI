## Documentation Policy

Use this after implementation or when generating docs.

## Include

- what changed
- why it changed
- how to use it
- commands
- configuration
- migration notes
- known limitations
- test result

## Rules

- Do not overclaim.
- Do not hide failed or skipped tests.
- Use concise technical writing.
- Prefer examples over abstract explanation.

## Output

```md
## Summary

## Usage

## Configuration

## Test result

## Migration notes

## Known limitations
```
