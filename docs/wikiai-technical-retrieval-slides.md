# WikiAI: технические нюансы RAG, BM25, lexical features и ColBERT

## Стиль

Рекомендуемый стиль: **Архитектурный**. Аудитория - эксперты предметной области и технические участники, поэтому слайды должны показывать не маркетинговые обещания, а управляемые режимы, tradeoff и эксплуатационные последствия.

## Слайды

### 1. WikiAI Retrieval: управляемая суперпозиция, а не один поиск
Тезис: качество RAG определяется тем, какие кандидаты попали в контекст до обращения к LLM.
- Vector/Qdrant ищет смысловую близость по embedding.
- BM25/FTS5 удерживает точные слова, термины, названия систем и регламентов.
- ColBERT уточняет релевантность на уровне token vectors.
- ACL и trust model фильтруют результаты до RAG-контекста.
- LLM получает только разрешенные chunks и источники.
Визуал: pipeline query -> vector/BM25/ColBERT -> ACL/trust -> RAG context -> LLM.

### 2. Как формируется RAG-контекст
Тезис: RAG - это не генерация ответа, а дисциплина отбора источников.
- Syncer режет страницы и вложения на chunks и отправляет их в Gateway.
- Gateway обновляет dense vector index, BM25/FTS5, optional trigram и optional ColBERT.
- При запросе берутся кандидаты из включенных retrieval-режимов.
- Затем применяются MediaWiki ACL, trust policy, лимиты `topK` и контекстные ограничения.
- В ответе показываются sources, чтобы эксперт видел происхождение знания.
Визуал: две дорожки: индексация страницы и обработка пользовательского запроса.

### 3. BM25 и lexical gate: когда точное слово сильнее смысла
Тезис: BM25 нужен, чтобы семантический поиск не уводил запрос от конкретных терминов.
- `simple_stem` снимает частые русские окончания и ищет по prefix: `кухня`, `кухню`, `кухней` -> `кухн*`.
- `lexicalMinMatchedTerms` отбрасывает chunks, где совпало слишком мало слов запроса.
- `lexicalGateMode=when_bm25_available` оставляет BM25-кандидатов, если они найдены.
- Vector score используется как дополнительный вес для тех же chunks, а не как повод вернуть semantic-only соседей.
- Если BM25 нашел только широкие совпадения, лучше пустая выдача, чем случайный RAG-контекст.
Визуал: таблица BM25 exact / BM25 broad / vector-only fallback.

### 4. Triangle retrieval: смысл, текст и late interaction
Тезис: разные методы закрывают разные типы ошибок поиска.
- Semantic vector хорошо ловит смысл, но может вернуть общий похожий документ.
- BM25 хорошо ловит буквальные термины, но хуже переносит опечатки и шум.
- ColBERT сравнивает запрос и документ детальнее через token-level MaxSim.
- `hybrid` смешивает vector и BM25; `hybrid_colbert` добавляет ColBERT rerank.
- `colbert_full` ищет первый набор кандидатов в отдельном ColBERT index.
Визуал: треугольник Semantic / Lexical / Late interaction с примерами запросов.

### 5. Experimental lexical features: editDistance и trigram на примерах
Тезис: typo tolerance включается ступенчато: сначала дешевый query-time режим, затем отдельный trigram index.
- `editDistance` не меняет индекс: Gateway добавляет укороченный prefix и проверяет, что найденный терм отличается не больше чем на одну правку.
- Пример `editDistance`: запрос `адмиристрирование систем` может подтянуть термин `администрирование`, если остальные слова подтверждают контекст.
- `editDistance` лучше для единичной опечатки в длинном слове: дешево, быстро, без backfill и без роста SQLite.
- `trigram` строит отдельный индекс из 3-символьных фрагментов и включается только если BM25 не дал пригодных chunks.
- Пример `trigram`: запрос `инфармацыонных систеи` может найти chunk `информационные системы`, потому что совпадает много 3-грамм.
Визуал: мини-таблица запрос -> BM25 -> editDistance -> trigram -> эффект.

### 6. ColBERT: варианты, цена и переключение моделей
Тезис: ColBERT повышает качество ранжирования, но требует отдельной эксплуатационной дисциплины.
- `hybrid_colbert`: сначала BM25/vector, затем ColBERT rerank уже разрешенных chunks.
- `colbert_full`: ColBERT index сам дает первый набор кандидатов до ACL/trust.
- Без ColBERT контур помечается `limited_ready`; для production он фактически обязателен.
- При смене модели нужен candidate index, complete build и promote.
- Active collection не перезаписывается при build, чтобы был rollback.
Визуал: две схемы hybrid_colbert и colbert_full плюс жизненный цикл candidate -> promote.

### 7. Карта выбора режимов для экспертов
Тезис: настройки включаются под класс запросов, а не "всё сразу".
- Термины, коды, названия систем: BM25 gate + `simple_stem`.
- Близкие формулировки и естественный язык: vector + BM25 hybrid.
- Единичные опечатки: `editDistance`, если нет роста шума на контрольных запросах.
- Сильные опечатки, OCR-шум, плохие запросы: `trigram` после backfill и staging benchmark.
- Production-контур: hybrid + ColBERT; trigram только при `readiness.passed=true`.
Визуал: decision matrix сценарий -> режим -> польза -> риск/проверка.

## Примечания для докладчика

- `editDistance` и `trigram` не надо объяснять как конкурентов ColBERT: это lexical fallback для других типов проблем.
- `trigram` не вызывает LLM, embeddings или ColBERT rebuild, но увеличивает SQLite/FTS storage и требует readiness.
- Если нужно уложиться в короткое выступление, слайды 3-5 являются центральными; слайды 1-2 вводят контекст, 6-7 фиксируют решения.
