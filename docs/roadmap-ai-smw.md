# Roadmap: Wiki AI + Semantic MediaWiki

Документ связывает два ТЗ в один управляемый план реализации:

- `TZ_MediaWiki_AI_Knowledge_Management_v1.2.md` остается базовым ТЗ продукта:
  AI-поиск, чат, проверка прав чтения, индексация, администрирование, тестовая корпоративная wiki.
- `TZ_SMW_AI_Integration_v1.2.md` остается специализированным дополнением:
  Semantic MediaWiki, свойства, формы, онтология, AI-разметка и векторизация
  семантических фактов.
- Этот roadmap не заменяет ТЗ, а задает порядок внедрения и приемочные ворота.

## Текущее состояние стенда

Проверено на текущем контейнере `mediawiki`:

| Область | Состояние |
|---------|-----------|
| MediaWiki | 1.45.3 |
| PHP | 8.3.31 |
| VisualEditor | установлен |
| AIAssistant | установлен |
| Semantic MediaWiki | установлен `7.0.0-alpha` / `dev-master` |
| VEForAll | установлен `0.8` |
| Page Forms | установлен `6.0.8` |
| Composer в контейнере | установлен `2.10.0` |
| `LocalSettings.php` | смонтирован извне read-only в контейнер |

Вывод: rollout на тестовом стенде выполнен, но SMW идет через dev-ветку, потому
что стабильный SMW 6.0.1 не совместим с MediaWiki 1.45.3. Для production нужен
стабильный релиз SMW с поддержкой MediaWiki 1.45.x или закрепленный проверенный
commit dev-ветки.

## Стратегия документов

Файлы `TZ_*` лучше не объединять физически:

- базовое ТЗ описывает продуктовую систему и уже достаточно большое;
- SMW/VEForAll является отдельным этапом с отдельными рисками миграции;
- раздельные файлы проще согласовывать, версионировать и принимать по этапам.

Общая навигация:

1. Базовый контур: `TZ_MediaWiki_AI_Knowledge_Management_v1.2.md`.
2. Семантический контур: `TZ_SMW_AI_Integration_v1.2.md`.
3. План внедрения и приемки: этот файл.
4. Операционный runbook: `docs/deployment-smw-veforall.md`.

## Этап 0. Compatibility audit

Цель: подтвердить, что текущая MediaWiki и PHP совместимы с выбранными версиями
SMW, Page Forms и VEForAll.

Работы:

- запустить `node scripts/audit-smw-rollout.mjs`;
- сверить версии расширений с официальными страницами MediaWiki/SMW;
- выбрать способ установки Composer-зависимостей: внутри контейнера, через
  отдельный disposable composer image или через build нового образа;
- сделать backup базы, `images/`, `LocalSettings.php` и `extensions/`;
- подготовить rollback-команду и критерии отката.

Приемка:

- есть отчет аудита;
- известен путь установки без ручных правок внутри контейнера;
- подтверждено, что текущая wiki продолжает работать после dry-run.

## Этап 1. Semantic MediaWiki base

Цель: включить хранилище семантических фактов без изменения AI-поиска.

Работы:

- установить Semantic MediaWiki совместимой версии;
- добавить настройки в управляемый `LocalSettings.php`;
- выполнить maintenance update и SMW setup/rebuild scripts;
- создать базовые свойства: департамент, отдел, тип документа, владелец,
  дата действия, статус, система, процесс;
- добавить несколько семантически размеченных страниц в тестовый корпоративный
  контент.
- для корпоративных namespace включить `$smwgNamespacesWithSemanticLinks`:
  `3000`, `3010`, `3020`, `3030`.

Приемка:

- служебные страницы SMW открываются;
- свойства видны на страницах;
- обычный поиск, чат и проверка прав чтения не деградировали;
- Syncer по-прежнему индексирует страницы с сервисной сессией.

## Этап 2. Page Forms + VEForAll

Цель: дать администраторам и владельцам контента VisualEditor-формы для
структурированной разметки страниц.

Работы:

- установить Page Forms;
- установить VEForAll;
- создать формы для регламентов, инструкций, FAQ и приказов;
- подключить VisualEditor-поля к формам;
- ограничить редактирование форм группами владельцев контента.

Приемка:

- пользователь может создать документ через форму без ручного wiki markup;
- VisualEditor работает внутри формы;
- права на чтение и редактирование не расширились;
- семантические свойства сохраняются и доступны SMW-запросами.

## Этап 3. AI ontology API

Цель: сделать семантические факты частью RAG-контура.

Статус на стенде:

- Syncer читает SMW facts и кладет их в `semantic_facts` payload Qdrant.
- проверка прав чтения для semantic payload принята на 34 страницах и 170 чанках.
- Добавлены admin endpoints Gateway:
  - `GET /api/admin/semantic/status`;
  - `GET /api/admin/semantic/search?property=...&value=...`.
- Добавлен UI-блок "Семантическая диагностика" в `Служебная:AI-администрирование`.
- Чат получает SMW facts как отдельный структурированный блок "Свойства
  документа" в prompt context.
- Добавлен запуск Syncer reindex из админки через Gateway -> Syncer job API;
  вложения по умолчанию выключены, OpenAI не используется.
- Осталось: browser acceptance админского UI под пользователем `sysop`/`aiadmin`
  и расширение UX для SMW maintenance/rebuild из админки.

Работы:

- расширить Syncer: читать SMW facts/properties вместе с текстом страницы;
- хранить в Qdrant payload семантические поля с теми же правами чтения, что и страница;
- добавить admin endpoint для статуса SMW-интеграции;
- добавить диагностический поиск по свойствам без LLM;
- обновить prompt builder: использовать свойства как структурированный контекст.

Приемка:

- поиск по департаменту, типу документа и процессу работает без LLM;
- чат цитирует источники и не использует факты из недоступных страниц;
- runtime-проверка прав чтения применяется к семантическому контексту так же, как к
  обычным чанкам.

## Этап 4. AI-assisted markup

Цель: помочь владельцам контента размечать документы, не увеличивая стоимость
OpenAI API на тестовом стенде.

Работы:

- сделать extraction pipeline opt-in;
- по умолчанию использовать локальные модели/эвристики для типов документов и
  базовых свойств;
- OpenAI/LiteLLM extraction запускать только по явному флагу и лимитам бюджета;
- сохранять AI-предложения как draft, требующий подтверждения человеком;
- добавить cost guards: лимит страниц, лимит токенов, dry-run, кеш результатов.

Приемка:

- тесты без OpenAI покрывают основной pipeline;
- live OpenAI-тесты вынесены в отдельный opt-in набор;
- AI не публикует семантические свойства без подтверждения, если это не разрешено
  настройкой стенда.

## Этап 5. Acceptance and rollout

Цель: принять semantic rollout без регрессии базового AI-контура.

Минимальные проверки:

- unit tests Syncer/Gateway;
- live-проверка прав чтения на корпоративном контенте;
- reindex protected namespaces с Syncer service-user credentials;
- проверка SMW pages/forms;
- проверка поиска без LLM;
- один ограниченный live LLM smoke test при наличии бюджета;
- secret scan/diff review перед коммитом.

Критерии готовности:

- расширения установлены воспроизводимо;
- rollback проверен документально;
- стоимость live OpenAI тестов ограничена флагами и малым набором документов;
- администрирование позволяет управлять видами документов, требующими
  распознавания;
- все новые semantic facts проходят через проверку прав чтения.
