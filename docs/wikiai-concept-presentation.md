# WikiAI: концепция презентации продукта

## Рекомендуемые стили PPTX

Пользователь попросил подготовить все три варианта, чтобы сравнить подачу:

- **Совет ИТ** - строгая версия для ИТ-директора, технического директора, ИБ, архитекторов и руководителей. Минимум декора, сильные заголовки, таблицы решений и управленческие выводы.
- **Архитектурный** - версия для архитектурного комитета и эксплуатации. Больше схем, потоков данных, границ ответственности и режимов работы.
- **Продуктовый** - версия для объяснения продукта и демонстрации. Больше пользовательских сценариев, карточек ценности и примеров применения AI.

Во всех версиях текст должен быть на русском языке. Без перевода оставляются только имена продуктов, протоколов, режимов и устойчивые технические термины: `WikiAI`, `MediaWiki`, `RAG`, `LLM`, `Qdrant`, `LiteLLM`, `ColBERT`, `BM25/FTS5`, `SSE`, `OIDC`, `MCP`, `API`, `webhook`, `cookie`, `namespace`, `Redis`, `SQLite`, `Postgres`, `Nginx`, `Docker`, `CI`.

## Главный тезис

WikiAI превращает корпоративную MediaWiki из пассивного хранилища документов в управляемую AI-платформу знаний: сотрудники получают поиск и чат по корпоративному контенту, а руководство сохраняет контроль над доступом, качеством источников, стоимостью AI и готовностью к промышленной эксплуатации.

## Рабочие допущения

- Продукт почти готов к промышленному внедрению, но финальная формулировка должна звучать как "готов к промышленному запуску после закрытия приемочного чек-листа", а не как "полностью введен в промышленную эксплуатацию".
- Главный акцент презентации - места использования AI и управляемость AI, а не глубокая реализация каждого API.
- Основная версия помещается в 18-20 слайдов. Термины и технические детали можно вынести в приложение.
- Готовность к промышленной эксплуатации нужно показывать честно: проверки состояния, CI, проверка секретов и журнал аудита есть; уровни диагностического режима `Basic`/`Verbose` и операционная доставка логов в несколько приемников по просмотренным файлам не подтверждены и должны быть отдельной P0-задачей перед жестким заявлением о готовности.

## Сюжетная линия

1. **Зачем продукт нужен:** корпоративные знания уже есть в MediaWiki, но ими трудно пользоваться быстро, безопасно и с учетом качества источников.
2. **Что меняется с WikiAI:** AI-поиск, AI-чат, прозрачные источники, модель доверия, семантические факты, администрирование и интеграции.
3. **Где используется AI:** векторные представления, RAG, LLM-ответ, выявление конфликтов, автозаполнение семантических полей, онтологические векторы, ColBERT. Отдельно показать, где AI не принимает решение: права доступа, политики, секреты, журнал аудита.
4. **Как система работает:** индексация, поиск, чат, проверка прав при выдаче ответа, админка, внешний API/MCP.
5. **Почему это управляемо:** закрытый контур размещения, MediaWiki как источник истины, запрет доступа при сомнении, LiteLLM как управляемый LLM-шлюз, отключаемые режимы AI.
6. **Что нужно для запуска:** приемка, стабильная версия или закрепленный commit SMW, учетные данные сервисного пользователя, наблюдаемость и логирование, резервное копирование и восстановление, решение по SQLite/Postgres.

## Структура слайдов

### 1. WikiAI: AI-платформа знаний поверх MediaWiki

**Назначение:** открыть презентацию управленческим тезисом.

**Ключевые сообщения:**
- WikiAI - AI-поиск и чат-ассистент для корпоративной MediaWiki в закрытом контуре.
- Система работает с учетом прав чтения пользователя и показывает источники ответа.
- Ценность не в "чат-боте", а в управляемом доступе к корпоративным знаниям.

**Фокус AI:** AI помогает найти и объяснить знания, но не заменяет систему прав MediaWiki.

**Визуал:** титульный слайд с простой схемой: "контент MediaWiki -> WikiAI -> сотрудники / эксперты / администраторы / внешние системы".

### 2. Почему текущей вики недостаточно

**Назначение:** зафиксировать организационную проблему.

**Ключевые сообщения:**
- Документы есть, но пользователи не всегда знают точные названия страниц и категории.
- Обычный поиск по ключевым словам плохо отвечает на вопросы естественным языком.
- Устаревшие, противоречивые и неофициальные материалы могут выглядеть одинаково.
- Закрытые документы нельзя просто отправить во внешний AI-сервис без риска утечки.

**Фокус AI:** AI нужен не ради генерации текста, а чтобы уменьшить время поиска и связать вопрос пользователя с релевантными фрагментами знаний.

**Визуал:** "до/после": слева ручной поиск по вики, справа вопрос естественным языком с ответом и источниками.

### 3. Что получает организация

**Назначение:** показать управленческую ценность для всей аудитории.

**Ключевые сообщения:**
- Сотрудники быстрее находят регламенты, инструкции, FAQ и внутренние решения.
- Руководители получают контролируемое распространение знаний без обхода прав доступа.
- Эксперты видят источники, оценку доверия, конфликты и кандидаты на обновление.
- Эксплуатация получает проверки состояния, админку, переиндексацию, резервное копирование, восстановление и CI-проверки.

**Фокус AI:** AI становится интерфейсом к знаниям, а не отдельной неконтролируемой базой.

**Визуал:** матрица "роль -> сценарий -> эффект".

### 4. Где именно используется AI

**Назначение:** сразу сфокусировать презентацию на AI-сценариях.

**Ключевые сообщения:**
- **Векторные представления (`embeddings`):** текст вики и запрос пользователя переводятся в векторы для смыслового поиска.
- **RAG:** LLM отвечает только после подбора разрешенных источников.
- **LLM через LiteLLM:** генерация ответа, потоковый чат, резервирование и маршрутизация на уровне LLM-шлюза.
- **Автозаполнение семантических полей:** AI помогает заполнить свойства документа с учетом режима работы и чувствительных полей.
- **Онтологические векторы:** свойства Semantic MediaWiki получают векторное представление для классификации и поиска похожих свойств.
- **ColBERT:** отдельный локальный слой нейросетевого ранжирования для более точного поиска по токенам.
- **Выявление конфликтов:** механизм показывает противоречия и низкое доверие к источникам.

**Важно:** ACL, секреты, журнал аудита, политики индексации и правила доверия не отдаются AI на самостоятельное решение.

**Визуал:** тепловая карта "функция продукта -> используется AI / не используется AI".

### 5. Пользовательский сценарий: AI-поиск

**Назначение:** показать основной пользовательский сценарий.

**Ключевые сообщения:**
- Пользователь задает вопрос естественным языком.
- Gateway проверяет MediaWiki-сессию, получает группы пользователя и ищет только доступные фрагменты.
- Результаты показывают страницы, фрагменты и ссылки на источники.
- Гибридный поиск объединяет смысловой поиск, BM25/FTS5 и опциональный ColBERT.

**Фокус AI:** векторные представления и нейросетевое ранжирование помогают найти смысловые совпадения, а BM25 удерживает точные слова запроса.

**Визуал:** экран или макет "вопрос -> результаты -> источник".

### 6. Пользовательский сценарий: AI-чат

**Назначение:** объяснить чат как продолжение поиска, а не отдельную память без контроля.

**Ключевые сообщения:**
- Каждый вопрос чата заново проходит через RAG и проверку прав.
- Ответ передается потоком через SSE, пользователь видит генерацию постепенно.
- Источники возвращаются вместе с ответом.
- История чатов хранится управляемо: активные чаты, архив, экспорт собственного архива.

**Фокус AI:** LLM формирует ответ на основе подобранных источников; история диалога не расширяет права доступа.

**Визуал:** последовательная схема "сообщение -> подбор источников -> LLM -> потоковый ответ -> источники".

### 7. RAG простыми словами

**Назначение:** объяснить базовый AI-механизм без перегруза.

**Ключевые сообщения:**
- **Embedding** - числовое представление смысла текста.
- **Векторная база данных** - хранилище таких числовых представлений; в WikiAI используется Qdrant.
- **RAG** - сначала найти релевантные источники, потом дать их LLM как контекст для ответа.
- **BM25/FTS5** - классический текстовый поиск по словам; не является AI.
- **ColBERT** - более детальный нейросетевой поиск и переранжирование на уровне токенов.

**Фокус AI:** генеративная модель отвечает, а retrieval-слой и классический поиск подбирают источники.

**Визуал:** три слоя: "текст -> подбор источников -> ответ LLM".

### 8. Принцип безопасности: права MediaWiki остаются источником истины

**Назначение:** снять ключевой риск ИБ и руководителей.

**Ключевые сообщения:**
- MediaWiki хранит контент, пользователей, сессии и права доступа.
- Gateway аутентифицирует пользователя через MediaWiki cookie/API.
- Qdrant хранит фрагменты с `allowed_groups`, но для сомнительных случаев Gateway делает проверку чтения через MediaWiki API в момент ответа.
- Модель безопасности консервативная: если доступ нельзя подтвердить, документ не показывается.
- Временно устаревший доступ после изменения группы допустим до истечения сессии или кеша; выдача документа пользователю, у которого никогда не было доступа, недопустима.

**Фокус AI:** AI видит только уже разрешенный контекст; AI не принимает решение о праве чтения.

**Визуал:** "шлюз ACL MediaWiki" перед результатами Qdrant и перед контекстом LLM.

### 9. Качество источников: модель доверия и конфликты

**Назначение:** показать, что продукт управляет надежностью знаний, а не только релевантностью.

**Ключевые сообщения:**
- Оценка доверия рассчитывается по явным правилам: `namespace`, категория, тег, группа автора, шаблон, свойства страницы, свежесть, семантические факты.
- Администратор настраивает модели доверия, сущности и правила через UI.
- Поиск и чат фильтруют контекст по политике доверия: черновики, устаревшие материалы, источники с низким доверием и проверенные источники обрабатываются по-разному.
- Выявление конфликтов помогает показать противоречие и рекомендуемый источник.

**Фокус AI:** AI помогает сформулировать ответ и обнаружить конфликт, но правила доверия явные и управляемые.

**Визуал:** шкала оценки доверия и пример "официальный документ против устаревшей wiki-страницы".

### 10. Semantic MediaWiki: переход от текста к структурированным знаниям

**Назначение:** объяснить SMW как усилитель AI, а не отдельную сложность.

**Ключевые сообщения:**
- SMW хранит структурированные факты: департамент, отдел, тип документа, владелец процесса, статус, система, процесс, критичность и другие свойства.
- Syncer индексирует семантические факты вместе с текстовыми фрагментами.
- Чат добавляет "Свойства документа" в контекст для LLM.
- Диагностический поиск по свойствам работает без LLM.
- ACL семантических фактов наследует ACL исходной страницы.

**Фокус AI:** структурированные факты делают ответы точнее, а онтологические векторы помогают классифицировать и связывать свойства.

**Визуал:** карточка документа: "текст + свойства + ACL + метаданные доверия".

### 11. AI-помощь в разметке и онтологические векторы

**Назначение:** показать расширенный сценарий AI для владельцев контента и экспертов.

**Ключевые сообщения:**
- Администратор задает семантические свойства и генерирует для них онтологические векторы через локальные векторные представления.
- При сохранении страницы AI может предложить заполнение пустых семантических полей.
- Для чувствительных свойств и автоприменения есть настройки, ограничения и журнал аудита.
- Подход снижает ручную разметку, но сохраняет контроль человека и пробный режим без записи.

**Фокус AI:** AI предлагает структуру и классификацию, человек или политика подтверждают применение.

**Визуал:** жизненный цикл "свойство -> вектор -> редактирование страницы -> предложение AI -> подтверждение/применение".

### 12. Индексация и переиндексация

**Назначение:** показать, как вики-контент попадает в AI-контур.

**Ключевые сообщения:**
- Расширение MediaWiki отправляет webhook при edit/delete/move/protect.
- Syncer получает страницу через MediaWiki API, извлекает текст, семантические факты и вложения по политике.
- Текст разбивается на фрагменты, получает векторные представления и записывается в Qdrant.
- Gateway обновляет BM25/FTS5 и опционально индекс ColBERT.
- Полная переиндексация нужна для первичной загрузки и восстановления после потерянных событий.

**Фокус AI:** векторные представления строятся на этапе индексации; LLM не нужен для обычной переиндексации.

**Визуал:** процесс "webhook/полная переиндексация -> загрузка страницы -> фрагменты -> векторы -> Qdrant/FTS/ColBERT".

### 13. Вложения и распознавание документов

**Назначение:** показать, что знания не ограничены текстовыми вики-страницами.

**Ключевые сообщения:**
- Файлы MediaWiki рассматриваются как страницы `File:`.
- Минимум индексируются метаданные файла.
- PDF/DOCX/XLSX и текстовые форматы могут идти в извлечение текста.
- Изображения могут идти в OCR; мультимодальная модель рассматривается как отключаемый будущий слой.
- Политика MIME управляет режимами: `text`, `ocr`, `metadata`, `disabled`.

**Фокус AI:** OCR/vision и последующая RAG-обработка расширяют доступ к знаниям, но включаются по политике и с учетом нагрузки.

**Визуал:** таблица MIME-типа и режима обработки.

### 14. Административная платформа

**Назначение:** показать, что продукт управляем изнутри MediaWiki.

**Ключевые сообщения:**
- `Служебная:AI-администрирование` доступна только `sysop`/`aiadmin`.
- Основные вкладки: обзор, сервисы, LLM, векторные представления, webhook, RAG/фрагментация, индексация, распознавание документов, доверие, хранение чатов, онтологические векторы, внешний API, журнал аудита.
- Секреты не показываются: только признак настройки.
- Безопасные проверки сервисов не создают и не меняют страницы.
- Журнал аудита фиксирует изменения административных настроек.

**Фокус AI:** поведение AI управляется через настройки: модель, RAG, обогащение, доверие, чувствительные свойства, ограничения стоимости.

**Визуал:** схема вкладок админки как панель управления.

### 15. Интеграции и границы ответственности

**Назначение:** показать архитектуру, понятную ИТ-директору, техническому директору, архитекторам и ИБ.

**Ключевые сообщения:**
- MediaWiki + AD/LDAP: контент, пользователи, группы и права.
- Nginx/reverse proxy: единый origin, `/api/*` направляется в Gateway, обычные маршруты вики остаются в MediaWiki.
- Gateway: поиск, чат, админка, внешний API, проверка ACL, контекст для LLM, потоковая выдача.
- Syncer: webhook, переиндексация, фрагменты, векторные представления, метаданные Qdrant, семантические факты.
- Qdrant: плотные векторы и отдельная коллекция ColBERT.
- Redis: группы пользователя, быстрый кеш выполнения и кеш истории чата.
- SQLite/Postgres: конфигурация админки, журнал аудита, профили, доверие, реестр чатов.
- LiteLLM: OpenAI-совместимый LLM-шлюз, маршрутизация, резервирование, лимиты запросов.
- Ollama/vLLM/OpenAI через LiteLLM: векторные представления и/или генерация по выбранной политике.
- Адаптер внешнего API/MCP: безопасный слой для сторонних систем через Gateway, без прямого доступа к Qdrant, Redis и MediaWiki.

**Фокус AI:** AI-провайдеры подключаются через управляемый шлюз и не становятся источником истины.

**Визуал:** блок-схема с границами ответственности и стрелками данных.

### 16. Режимы и вариативность

**Назначение:** показать, что продукт можно адаптировать без переписывания кода.

**Ключевые сообщения:**
- Режим поиска: `vector_only`, `hybrid`, `colbert_full`, `hybrid_colbert`.
- ColBERT можно выключить, использовать как отдельный индекс или как переранжирование.
- LLM можно менять через alias в LiteLLM без изменения Gateway.
- Семантические факты, вложения, OCR, AI-обогащение и автозаполнение имеют включаемые режимы.
- Внешний API поддерживает MediaWiki cookie, OIDC Bearer и ограниченный анонимный поиск, если это включено.
- Хранение: SQLite для разработки, теста и пилота; Postgres при SLA, HA, комплаенсе, аудите или нескольких инстансах.

**Фокус AI:** AI-возможности включаются дозированно: от локальных векторных представлений до LLM-обогащения и внешних интеграций.

**Визуал:** матрица переключателей "режим -> эффект -> риск/ограничение".

### 17. Нелинейные процессы и исключения

**Назначение:** показать зрелость для реальной эксплуатации.

**Ключевые сообщения:**
- Переиндексация защищенного `namespace` блокируется, если нет учетных данных сервисного пользователя MediaWiki.
- Если проверка чтения не подтверждает доступ, фрагмент отбрасывается.
- Если LLM недоступен, система может вернуть деградированный ответ с источниками, если они найдены.
- Если ColBERT недоступен, режим `fallback_current` возвращает текущую гибридную выдачу; режим `fail_search` делает ошибку явной.
- `dryRun=true` позволяет проверять переиндексацию, доверие и автозаполнение без записи.
- OpenAI/LiteLLM smoke-тесты запускаются вручную и только по явному включению, чтобы не тратить платный API автоматически.
- AI-автозаполнение не должно создавать бесконечную петлю: правка сервисного пользователя определяется и обрабатывается отдельно.

**Фокус AI:** критические ветки имеют резервный путь, пробный режим без записи или явное отключение AI.

**Визуал:** карта процесса с ветками "успех / нет доступа / нет LLM / пробный режим / резервный путь".

### 18. Готовность к промышленной эксплуатации: что уже есть и что надо закрыть

**Назначение:** честно показать состояние почти готового продукта.

**Ключевые сообщения:**
- Есть Docker Compose для Gateway, Syncer, Qdrant, Ollama, ColBERT и локальная схема стенда.
- Есть endpoint проверки состояния Gateway, Syncer, Qdrant, ColBERT и состояние сервисов в админке.
- Есть GitLab CI: validate, test, typecheck, build, docker build, проверка секретов.
- Есть Docker-образы Gateway/Syncer без root-пользователя и поставочный артефакт расширения MediaWiki.
- Есть эксплуатационная инструкция, руководство развертывания, приемочный чек-лист, процедуры резервного копирования и восстановления.
- Есть журнал аудита административных изменений и маскирование секретов в UI/API.
- Нужно закрыть перед жестким заявлением о готовности: приемка в браузере, стабильная совместимость SMW или политика закрепленного commit, учетные данные сервисного пользователя, baseline `npm audit`, решение SQLite/Postgres для SLA, уровни диагностики `Basic`/`Verbose`, структурированное логирование через stdout/stderr плюс операционный приемник логов.

**Фокус AI:** готовность к промышленной эксплуатации включает не только качество модели, но и управляемость, стоимость, диагностику и восстановление.

**Визуал:** таблица готовности "готово / требует решения / ответственный".

### 19. Данные, хранение и восстановление

**Назначение:** дать архитекторам и эксплуатации ясную картину зон с состоянием.

**Ключевые сообщения:**
- Компоненты с состоянием: БД MediaWiki, загруженные файлы, коллекции Qdrant, Redis, SQLite/Postgres, кеш моделей и индекс ColBERT.
- Компоненты без критического состояния: Gateway, Syncer, кодовый артефакт расширения MediaWiki, MCP adapter.
- Минимальный состав резервного копирования: БД MediaWiki, загруженные файлы, Qdrant `wiki_chunks`, коллекция ColBERT при использовании, административная БД, настройки Redis, если они еще используются.
- Полная переиндексация - рабочий путь восстановления поискового индекса, если MediaWiki является источником истины.
- DR/BCP требует определить RPO/RTO, частоту резервного копирования, проверки восстановления и поведение при недоступности LLM, Qdrant или MediaWiki.

**Фокус AI:** AI-индексы восстановимы из MediaWiki, но операционное восстановление должно быть проверено до промышленного запуска.

**Визуал:** карта состояния "источник истины / кеш / производный индекс / внешняя зависимость".

### 20. Решения для запуска и следующие шаги

**Назначение:** завершить презентацию списком решений.

**Ключевые сообщения:**
- Утвердить формат внедрения: пилот, ограниченное внедрение или полный промышленный запуск.
- Зафиксировать политику LLM: локальная LLM, OpenAI через LiteLLM, гибридный режим, бюджет живых проверок.
- Утвердить ACL и модель сервисного пользователя для защищенных `namespace`.
- Выбрать целевое хранилище: SQLite для пилота или Postgres для SLA/комплаенса.
- Утвердить базовый уровень наблюдаемости: диагностические режимы, структурированные логи, приемник логов, срок хранения журнала аудита.
- Пройти приемочный чек-лист на стенде и зафиксировать результаты.
- Подготовить короткий сценарий демонстрации: поиск, чат, источники, конфликт доверия, состояние админки, пробная переиндексация, онтологический вектор.

**Фокус AI:** финальное решение не "включить AI", а "включить управляемую AI-платформу знаний с понятными границами".

**Визуал:** чек-лист решений с ответственными и статусами.

## Приложение: термины для простого объяснения

| Термин | Простое объяснение |
| --- | --- |
| RAG | Подход, где система сначала находит источники, а потом просит LLM ответить на их основе. |
| Embedding | Числовое представление смысла текста. Похожие тексты получают похожие векторы. |
| Векторная база данных | База, которая быстро ищет похожие векторы; в WikiAI используется Qdrant. |
| BM25 / FTS5 | Классический поиск по словам и их совпадениям; не является AI. |
| ColBERT | Нейросетевой поиск и переранжирование, которое сравнивает запрос и документ подробнее, на уровне токенов. |
| LLM | Большая языковая модель, которая формирует текст ответа. |
| LiteLLM | Прокси-шлюз к LLM: единый API, маршрутизация, резервирование, лимиты запросов и аудит на уровне LLM-инфраструктуры. |
| Semantic MediaWiki | Расширение MediaWiki, позволяющее хранить факты как свойства страниц. |
| Онтологические векторы | Векторные представления семантических свойств, чтобы находить близкие свойства и классифицировать фрагменты. |
| Оценка доверия | Настраиваемая оценка надежности источника по явным правилам. |
| Webhook | Событие от MediaWiki в Syncer при изменении страницы. |
| SSE | Server-Sent Events; способ потоково передавать ответ ассистента в браузер. |
| OIDC Bearer | Токен внешней системы или пользователя для доступа к внешнему API. |
| MCP adapter | Адаптер, через который внешние AI-инструменты могут использовать Gateway API без прямого доступа к внутренним базам. |
| Компонент без состояния | Сервис не хранит критические данные и может быть пересоздан. |
| Компонент с состоянием | Компонент хранит данные и требует резервного копирования и восстановления. |
