# WikiAI: источник слайдов для PPTX

## Варианты стиля

- **Совет ИТ:** строгая версия для ИТ-директора, технического директора, ИБ и руководителей. Главный акцент - управленческая ценность, риски, готовность и решения.
- **Архитектурный:** версия для архитектурного комитета. Главный акцент - блоки системы, потоки данных, границы ответственности, режимы и восстановление.
- **Продуктовый:** версия для объяснения продукта. Главный акцент - пользовательские сценарии, места применения AI, демонстрационный путь и измеримая польза.

## Слайды

### 1. WikiAI: AI-платформа знаний поверх MediaWiki
Тезис: WikiAI превращает корпоративную MediaWiki в управляемую AI-платформу знаний.
- AI-поиск и чат-ассистент работают поверх существующей MediaWiki.
- Ответы строятся с учетом прав чтения пользователя.
- Система показывает источники и сохраняет управляемость корпоративных знаний.
Визуал: MediaWiki -> WikiAI -> сотрудники, эксперты, администраторы, внешние системы.

### 2. Почему текущей вики недостаточно
Тезис: знания есть, но пользователю трудно быстро найти точный и надежный ответ.
- Пользователь не всегда знает название страницы или категорию.
- Поиск по ключевым словам плохо работает с вопросами естественным языком.
- Устаревшие и неофициальные материалы могут выглядеть как достоверные.
- Закрытый контент нельзя безопасно отправлять во внешний AI-сервис.
Визуал: сравнение ручного поиска и вопроса естественным языком.

### 3. Что получает организация
Тезис: продукт снижает время поиска и повышает управляемость корпоративных знаний.
- Сотрудники быстрее находят регламенты, инструкции и FAQ.
- Руководители сохраняют контроль над доступом и качеством источников.
- Эксперты видят конфликты, оценку доверия и кандидаты на обновление.
- Эксплуатация получает админку, проверки состояния и процедуры восстановления.
Визуал: матрица роль -> сценарий -> эффект.

### 4. Где именно используется AI
Тезис: AI используется точечно и управляемо, а не вместо правил доступа и политик.
- Векторные представления помогают искать смысловые совпадения.
- RAG подбирает источники до генерации ответа LLM.
- LiteLLM управляет LLM-маршрутизацией и резервированием.
- AI помогает в семантической разметке и выявлении конфликтов.
- ColBERT усиливает точность нейросетевого ранжирования.
Визуал: карта функций, где AI используется и где решения остаются за правилами.

### 5. Пользовательский сценарий: AI-поиск
Тезис: пользователь задает вопрос, система возвращает доступные источники.
- Gateway проверяет MediaWiki-сессию и группы пользователя.
- Qdrant ищет только по разрешенным фрагментам.
- BM25/FTS5 удерживает точные термины запроса.
- ColBERT может улучшить порядок результатов.
Визуал: вопрос -> проверка прав -> поиск -> результаты с источниками.

### 6. Пользовательский сценарий: AI-чат
Тезис: чат не обходит права доступа и заново подбирает контекст для каждого вопроса.
- Каждый вопрос проходит через RAG и проверку прав.
- Ответ передается потоком через SSE.
- Источники возвращаются вместе с ответом.
- История чатов управляется через активные и архивные сессии.
Визуал: сообщение -> подбор источников -> LLM -> потоковый ответ -> источники.

### 7. RAG простыми словами
Тезис: LLM отвечает не из памяти, а на основе найденных источников.
- Embedding переводит смысл текста в числа.
- Qdrant ищет похожие векторы.
- BM25/FTS5 добавляет точное совпадение слов.
- RAG передает LLM только разрешенный контекст.
Визуал: текст -> векторный поиск -> контекст -> ответ LLM.

### 8. Права MediaWiki остаются источником истины
Тезис: AI не решает, кому можно читать документ.
- MediaWiki хранит пользователей, группы, сессии и права.
- Gateway проверяет пользователя через MediaWiki cookie/API.
- Сомнительные результаты проходят проверку чтения через MediaWiki API.
- Если доступ нельзя подтвердить, документ не показывается.
Визуал: ACL-шлюз MediaWiki перед поисковой выдачей и LLM-контекстом.

### 9. Качество источников: модель доверия и конфликты
Тезис: система различает официальный, устаревший, черновой и спорный контент.
- Оценка доверия считается по явным правилам.
- Учитываются namespace, категория, теги, автор, шаблон и свежесть.
- Поиск и чат могут ограничивать контекст по политике доверия.
- Выявление конфликтов показывает спорные источники и рекомендуемый источник.
Визуал: шкала оценки доверия и пример конфликта источников.

### 10. Semantic MediaWiki: от текста к структурированным знаниям
Тезис: семантические факты делают ответы точнее и проверяемее.
- SMW хранит свойства страниц: департамент, процесс, статус, владелец.
- Syncer индексирует свойства вместе с текстовыми фрагментами.
- Чат добавляет свойства документа в контекст для LLM.
- ACL семантических фактов наследует ACL страницы.
Визуал: карточка документа: текст, свойства, ACL, доверие.

### 11. AI-помощь в разметке и онтологические векторы
Тезис: AI снижает ручной труд владельцев контента, но сохраняет контроль человека.
- Администратор задает семантические свойства.
- Система строит онтологические векторы через локальные embeddings.
- AI предлагает заполнить пустые свойства при сохранении страницы.
- Чувствительные свойства и автоприменение управляются политиками.
Визуал: свойство -> вектор -> предложение AI -> подтверждение.

### 12. Индексация и переиндексация
Тезис: AI-контур строится из актуального контента MediaWiki.
- Расширение MediaWiki отправляет webhook при изменениях страниц.
- Syncer загружает страницу, свойства и вложения по политике.
- Текст делится на фрагменты и записывается в Qdrant.
- Полная переиндексация нужна для первичной загрузки и восстановления.
Визуал: webhook/переиндексация -> страница -> фрагменты -> Qdrant/FTS/ColBERT.

### 13. Вложения и распознавание документов
Тезис: знания могут извлекаться не только из вики-страниц, но и из файлов.
- Файлы MediaWiki рассматриваются как страницы `File:`.
- Для файлов всегда можно индексировать метаданные.
- PDF и офисные документы могут идти в извлечение текста.
- Изображения могут идти в OCR, если это разрешено политикой.
Визуал: таблица политики MIME: `text`, `ocr`, `metadata`, `disabled`.

### 14. Административная платформа
Тезис: AI-поведение управляется из MediaWiki, а не через ручные правки кода.
- Админка доступна только sysop или aiadmin.
- Управляются сервисы, LLM, embeddings, RAG, индексация, доверие и чаты.
- Секреты маскируются и не показываются в UI/API.
- Безопасные проверки не создают и не меняют страницы.
Визуал: вкладки админки как панель управления.

### 15. Интеграции и границы ответственности
Тезис: каждый компонент отвечает за свою часть, а AI-провайдеры не становятся источником истины.
- MediaWiki и AD/LDAP отвечают за контент, группы и права.
- Gateway отвечает за поиск, чат, внешний API и проверку доступа.
- Syncer отвечает за индексацию и обновление фрагментов.
- LiteLLM отвечает за LLM-маршрутизацию и резервирование.
- Qdrant, Redis и SQLite/Postgres хранят разные типы состояния.
Визуал: блок-схема компонентов и стрелки данных.

### 16. Режимы и вариативность
Тезис: возможности включаются дозированно под требования стенда и безопасности.
- Режим поиска: vector_only, hybrid, colbert_full, hybrid_colbert.
- ColBERT можно выключить или использовать для переранжирования.
- Семантические факты, OCR и автозаполнение включаются политиками.
- SQLite подходит для пилота, Postgres нужен при SLA и нескольких инстансах.
Визуал: матрица режим -> эффект -> ограничение.

### 17. Нелинейные процессы и исключения
Тезис: критические ветки имеют явные проверки, пробный режим или резервный путь.
- Защищенные namespace не индексируются без сервисного пользователя.
- Если проверка чтения не проходит, фрагмент отбрасывается.
- Если LLM недоступен, можно вернуть деградированный ответ с источниками.
- dryRun позволяет проверять операции без записи.
Визуал: карта процесса с ветками успеха, отказа доступа и деградации.

### 18. Готовность к промышленной эксплуатации
Тезис: продукт близок к промышленному запуску, но финальные P0-пункты нужно закрыть честно.
- Есть Docker Compose, проверки состояния, CI и проверка секретов.
- Есть админка, журнал аудита, инструкции развертывания и эксплуатации.
- Есть поставочный артефакт расширения MediaWiki.
- Нужно закрыть приемку UI, SMW-совместимость, сервисного пользователя и P0-логирование.
Визуал: таблица готовности: готово, требует решения, ответственный.

### 19. Данные, хранение и восстановление
Тезис: MediaWiki остается источником истины, AI-индексы можно восстановить.
- Состояние хранят БД MediaWiki, загруженные файлы, Qdrant, Redis и административная БД.
- Gateway и Syncer можно пересоздавать как сервисы без критического состояния.
- Полная переиндексация восстанавливает поисковый индекс из MediaWiki.
- Для DR/BCP нужны RPO/RTO и проверенные процедуры восстановления.
Визуал: карта состояния: источник истины, кеш, производный индекс, зависимость.

### 20. Решения для запуска
Тезис: решение о запуске должно фиксировать не только AI-модель, но и контур управления.
- Утвердить формат внедрения: пилот, ограниченный запуск или промышленный запуск.
- Зафиксировать LLM-политику и бюджет живых проверок.
- Утвердить ACL и сервисного пользователя для защищенных namespace.
- Выбрать SQLite или Postgres.
- Закрыть наблюдаемость, логи и приемочный чек-лист.
Визуал: чек-лист решений с ответственными и статусами.
