## 1. ОБЩИЕ ПОЛОЖЕНИЯ

### 1.1 Цель
Разработка расширения MediaWiki, добавляющего:
- AI-поиск по содержимому вики (RAG-архитектура)
- AI-чат-ассистента с доступом к вики-контенту
- **Явную настраиваемую модель доверия** (сущности -> правила -> trust score)
- Административный интерфейс для конфигурации всей системы
- **Отключаемые блоки обработки персональных данных**
- **Права, связанные со стандартными группами MediaWiki**
- **Управление хранением и архивированием чатов**

### 1.2 Ключевые принципы
1. **Прозрачность источников** -- пользователь всегда видит, откуда взята информация
2. **Явная модель доверия** -- сущности, правила и веса задаются в UI, не hardcoded
3. **Градуированное доверие** -- разные сущности имеют разный вес, настраивается админом
4. **Контролируемая автономия LLM** -- админ настраивает, когда LLM может синтезировать, а когда должен делегировать человеку
5. **Отключаемость ПДн** -- блоки обработки персональных данных могут быть полностью выключены
6. **Стандартные права MediaWiki** -- интеграция с существующей системой прав
7. **Управляемое хранение чатов** -- админ задает правила, пользователь контролирует свою историю

#### A.1 Компоненты интерфейса пользователя

### 2.2 Блок B: AI-Чат (AI Chat Assistant)

#### B.1 Архитектура диалога
- **Контекстный поиск** -- каждый вопрос пользователя проходит через RAG
- **Мультитуровая память** -- чат помнит предыдущие источники и противоречия
- **Каскадное уточнение** -- при противоречиях система задает уточняющие вопросы

#### B.2 Поведение при противоречиях в диалоге

**Пример диалога:**

Пользователь: А для руководителей?

    • HR Policy v3.2 (раздел 4.2): "Руководители подразделений

- Отправка в Slack/email/тикет-систему (настраивается админом)
- Трекинг: сколько раз источник вызывал сомнения

### 2.3 Блок C: Система Trust Score (Score Engine)

#### C.1 Концепция: Явная модель доверия через сущности

**Сущность (Entity)** -- любой объект MediaWiki, к которому можно применить правило доверия. Сущности задаются в UI администратором и формируют **модель доверия** -- набор правил, явно определяющих, как вычисляется trust score.

**Типы сущностей (заданы в UI):**

| Сущность | Описание | Примеры значений |
|----------|----------|------------------|
| **Namespace** | Пространство имен | `HR`, `Legal`, `Main`, `Draft` |
| **Category** | Категория страницы | `Официальные документы`, `Процедуры`, `Черновики` |
| **Tag** | Тег/метка страницы | `verified`, `deprecated`, `community` |
| **Author Group** | Группа автора | `sysop`, `bureaucrat`, `editor`, `user` |
| **Page Property** | Свойство страницы | `is_protected`, `is_redirect`, `has_review` |
| **Template** | Используемый шаблон | `Template:Official`, `Template:Draft` |
| **Content Type** | Тип содержимого | `text`, `table`, `infobox` |
| **Date Property** | Временной признак | `last_edit`, `creation_date` |

**Правило (Rule)** -- условие "Если сущность = X, то trust modifier = Y".

**Пример модели доверия (заданной в UI):**
```
Модель доверия "Корпоративная вики v1"

1. Namespace -> Базовый модификатор
   ├─ HR = +40 баллов
   ├─ Legal = +50 баллов
   ├─ Compliance = +50 баллов
   ├─ Main = +20 баллов
   ├─ Draft = -30 баллов (или исключить из индекса)
   └─ User = +10 баллов

2. Category -> Дополнительный модификатор
   ├─ Официальные документы = +20 баллов
   ├─ Процедуры = +15 баллов
   ├─ Политики = +25 баллов
   ├─ Черновики = -20 баллов
   └─ Архив = -40 баллов

3. Tag -> Статус верификации
   ├─ verified = +15 баллов
   ├─ reviewed = +10 баллов
   ├─ deprecated = -50 баллов + флаг OUTDATED
   ├─ community = +0 баллов
   └─ official = +20 баллов

4. Author Group -> Репутация автора
   ├─ sysop = +20 баллов
   ├─ bureaucrat = +25 баллов
   ├─ editor (с правом edittrusted) = +15 баллов
   ├─ user = +5 баллов
   └─ anonymous = +0 баллов

5. Page Property -> Технические признаки
   ├─ is_protected = +10 баллов
   ├─ has_review = +10 баллов
   ├─ is_redirect = -10 баллов
   └─ has_infobox = +5 баллов

6. Date Property -> Свежесть
   ├─ last_edit < 1 месяц = +15 баллов
   ├─ last_edit < 6 месяцев = +10 баллов
   ├─ last_edit < 1 год = +5 баллов
   ├─ last_edit < 2 года = +0 баллов
   ├─ last_edit > 2 года = -15 баллов + флаг OUTDATED
   └─ last_edit > 3 года = -30 баллов + флаг ARCHIVED

7. Template -> Шаблоны качества
   ├─ Template:Official = +20 баллов
   ├─ Template:Verified = +15 баллов
   ├─ Template:Draft = -20 баллов
   └─ Template:Deprecated = -50 баллов + флаг OUTDATED

8. Content Type -> Тип контента
   ├─ has_infobox = +5 баллов
   ├─ has_table = +3 баллов
   └─ has_gallery = +0 баллов

Формула: Trust Score = 50 (база) + SUM модификаторов сущностей
Ограничение: min = 0, max = 100
```

#### C.2 Алгоритм расчета Trust Score

```
Функция calculateTrustScore(page_id):
    base_score = 50  // Базовое значение, настраивается админом

    // Применяем правила в порядке приоритета (заданном в UI)
    for each rule in active_rules (ordered by priority):
        if rule.matches(page_id):
            base_score += rule.modifier

    // Применяем ограничения
    trust_score = clamp(base_score, 0, 100)

    // Определяем флаги
    flags = []
    if hasRule(page_id, 'deprecated') or isOlderThan(2 years):
        flags.push('OUTDATED')
    if hasRule(page_id, 'draft'):
        flags.push('DRAFT')
    if hasRule(page_id, 'official'):
        flags.push('OFFICIAL')
    if hasRule(page_id, 'verified'):
        flags.push('VERIFIED')

    return { score: trust_score, flags: flags, applied_rules: [...] }
```

#### C.3 Детекция противоречий

**Алгоритм Conflict Detection:**
1. При ретриве собираются топ-K чанков по семантической близости
2. LLM-промпт анализирует извлеченные факты на предмет противоречий
3. Если обнаружено противоречие:
   - Сравниваются trust scores источников
   - Приоритет отдается источнику с высшим score
   - Если scores близки (разница < порога, настраивается в UI) -- флаг "Требуется человеческое разрешение"
   - В ответе явно указывается конфликт

**Типы противоречий:**
- **Direct contradiction** -- прямое противоречие фактов
- **Temporal conflict** -- разные версии одного документа во времени
- **Scope conflict** -- факты верны в разных контекстах (например, разные подразделения)
- **Authority conflict** -- один источник официальный, другой -- нет

## 6. НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 6.1 Производительность
- Время ответа AI-поиска: < 3 секунд (p95)
- Время ответа AI-чата: < 5 секунд (p95)
- Индексация новой страницы: < 1 минута
- Пересчет trust score: асинхронно, < 5 минут для всей базы

### 6.2 Масштабируемость
- Поддержка до 100 000 страниц в индексе
- До 1000 одновременных пользователей
- Кэширование trust scores и частых запросов

### 6.3 Безопасность
- LLM никогда не получает персональные данные без анонимизации (если включено)
- Все запросы логируются для аудита (если включено)
- Админ-доступ только для групп с правами `ai-admin-dashboard`
- API-ключи LLM хранятся в зашифрованном виде

### 6.4 Отказоустойчивость
- Fallback при недоступности LLM: поиск по ключевым словам + показ источников
- Graceful degradation: если trust score недоступен -- использовать дефолт
- Кэш последних ответов на случай сбоев

### Приложение A: Примеры промптов для LLM (начальные примеры)

**Промпт: Conflict Detection & Resolution (CONFLICT_DETECTION)**
```
Ты -- анализатор корпоративной базы знаний. 

ПРАВИЛА РАБОТЫ С ИСТОЧНИКАМИ:
1. Каждый источник имеет trust score (0-100) и метаданные
2. При конфликте приоритет источнику с высшим trust score
3. Если разница trust < 15 -- отметить "требуется уточнение"
4. Всегда явно указывать наличие конфликта
5. Устаревшие источники (OUTDATED) понижаются на 30 баллов

ФОРМАТ ОТВЕТА:
- Прямой ответ на вопрос
- Если есть конфликт: блок "⚠️ Обнаружено противоречие"
- Для каждого источника: цитата + trust score + статус
- Рекомендация на основе приоритетного источника

ЗАПРЕЩЕНО:
- Скрывать наличие конфликта
- Синтезировать информацию без указания источника
- Игнорировать метаданные trust score
```

**Промпт: Low Confidence Response (LOW_CONFIDENCE)**
```
Если confidence < 70 или средний trust score источников < 50:

Не давай прямого ответа. Вместо этого:
"По вашему запросу найдена информация, но ее достоверность 
не позволяет дать уверенный ответ. 

Найденные источники:
[список с trust scores]

Рекомендации:
- Уточните запрос
- Свяжитесь с [отдел/эксперт]
- Проверьте актуальность в [ссылка на раздел]"
```

**Промпт: Answer Synthesis (ANSWER_SYNTHESIS)**
```
На основе следующих источников с их метаданными:
{sources}

Ответь на вопрос: {query}

Требования:
1. Всегда указывай источники с trust score
2. Если есть конфликт -- явно его опиши
3. Если источник устарел -- предупреди
4. Не добавляй информацию, отсутствующую в источниках
```
