# Требования проекта Wiki AI

Этот документ фиксирует требования, заложенные в текущих проектных материалах:
`README.md`, `mediawiki-ai-search-architecture.md`, `.env.example` и
`docker-compose.yml`. Это извлеченная спецификация намерений проекта; фактическая
реализация может отставать от отдельных пунктов.

## Назначение

Wiki AI должен предоставлять self-hosted AI-поиск и чат-ассистент поверх
корпоративной MediaWiki.

Основная цель: дать пользователям семантический поиск и ответы по содержимому
вики без нарушения прав доступа.

## Контур размещения

- Система должна работать в закрытом контуре без облачных сервисов.
- MediaWiki, Redis, LiteLLM, Nginx и часть LLM-инфраструктуры считаются внешними
  уже развернутыми сервисами.
- Компоненты проекта: AI Gateway, Syncer, MediaWiki Extension, React UI и Qdrant.
- Qdrant хранит векторный индекс и метаданные фрагментов.
- Ollama используется для эмбеддингов; vLLM и Ollama могут использоваться как
  локальные inference-движки через LiteLLM.
- Gateway и MediaWiki должны работать за одним reverse proxy и на одном домене,
  чтобы cookie сессии MediaWiki доходили до Gateway.
- Reverse proxy должен направлять `/api/*` в AI Gateway; обычные маршруты
  MediaWiki остаются в MediaWiki.

## Компоненты и ответственность

### MediaWiki

- Хранит контент, пользователей, сессии и права доступа.
- Интегрируется с Active Directory через PluggableAuth, LDAPProvider,
  LDAPAuthentication2 и LDAPGroups.
- Передает события изменения страниц в Syncer через hooks расширения.
- Не отвечает за AI-поиск, LLM-генерацию и векторную базу.
- Для тестового стенда может подключать конфиг `corporate-test-settings.php`,
  который создает корпоративные namespaces, группы и page-level ACL rules.

### Active Directory / LDAP

- Является источником групп пользователей.
- Должен поддерживать вложенные группы через transitive membership или
  эквивалентную настройку `nestedgroups: true`.

### MediaWiki Extension

- Регистрирует `Special:AIAssistant` для пользовательского поиска и чата.
- Регистрирует `Special:AIAdmin` для администрирования.
- Загружает React-приложение через ResourceLoader.
- Отправляет webhooks в Syncer при сохранении, удалении, перемещении и изменении
  защиты страницы.
- Должно оставаться тонким слоем без сложной бизнес-логики.

### AI Gateway

- Предоставляет API поиска и чата.
- Аутентифицирует пользователя через cookie MediaWiki и MediaWiki API.
- Получает группы пользователя и кеширует их в Redis.
- Создает embedding запроса через Ollama.
- Выполняет поиск в Qdrant с фильтрацией по ACL.
- Выполняет runtime-проверку доступа через MediaWiki API для сомнительных
  результатов.
- Формирует RAG-контекст и промпт для LLM.
- Отправляет запросы генерации в LiteLLM.
- Возвращает чат-ответы через Server-Sent Events.
- Хранит историю чата в Redis, привязанную к сессии пользователя.
- Предоставляет health/status API для проверки зависимостей.

### Syncer

- Получает webhook-события от MediaWiki.
- Получает содержимое страниц через MediaWiki API.
- Разбивает текст на чанки с настраиваемым размером и overlap.
- Получает embeddings через Ollama.
- Записывает чанки, vectors и payload metadata в Qdrant.
- Определяет ACL чанков минимум на уровне namespace.
- Для закрытых namespace должен поддерживать сервисную MediaWiki-сессию через
  `MW_SYNC_COOKIE`, чтобы индексировать защищенный контент без раскрытия паролей.
- Должен поддерживать полную переиндексацию как защиту от потерянных событий.
- Должен поддерживать backup/restore индекса Qdrant.

### Semantic MediaWiki / Page Forms / VEForAll

- Semantic MediaWiki является отдельным этапом расширения базового AI-контура.
- SMW должен хранить структурированные факты страниц: департамент, отдел, тип
  документа, владелец процесса, статус, систему, процесс и сроки действия.
- Page Forms должен предоставлять формы создания и редактирования регламентов,
  инструкций, FAQ и приказов.
- VEForAll должен позволять использовать VisualEditor внутри форм, чтобы
  владельцы контента не редактировали wiki markup вручную.
- Семантические свойства должны индексироваться Syncer как structured payload
  вместе с текстовыми чанками.
- ACL семантических фактов должен наследовать ACL исходной страницы; факт из
  закрытой страницы не должен попадать в поиск или чат пользователя без прав.
- AI-разметка семантических свойств должна быть opt-in и проходить через
  подтверждение владельцем контента, если стенд явно не разрешает автопубликацию.
- Live OpenAI/LiteLLM extraction tests должны быть отдельным opt-in набором с
  лимитом страниц и токенов; базовые semantic tests должны проходить без платных
  API.

### Qdrant

- Хранит vectors и payload фрагментов.
- Payload должен содержать как минимум `page_id`, `title`, `namespace`, `text`,
  `allowed_groups`, `chunk_index`, `total_chunks` и `last_modified`.
- Поиск должен фильтроваться по пересечению `allowed_groups` с группами
  пользователя.

### LiteLLM

- Должен быть единым OpenAI-compatible proxy для LLM-запросов.
- Должен позволять менять upstream-модели без изменения Gateway.
- Должен обеспечивать routing, fallback, rate limiting, кеширование и аудит, если
  это включено в инфраструктуре.

### Redis

- Хранит краткоживущий кеш групп пользователя.
- Хранит историю чата по сессии и conversation id.
- Может использоваться для runtime-настроек Gateway.
- Хранит управляемую политику распознавания документов для Syncer.

## Функциональные требования

### Поиск

- Пользователь открывает `Special:AIAssistant`.
- UI отправляет запрос в Gateway.
- Gateway проверяет MediaWiki-сессию пользователя.
- Gateway получает группы пользователя.
- Gateway строит embedding запроса.
- Gateway ищет релевантные чанки в Qdrant.
- В режиме `hybrid` Gateway должен дополнительно искать lexical-кандидатов через SQLite FTS5/BM25, применять эвристику русских окончаний и не подмешивать semantic-only кандидатов, если BM25 дал пригодные совпадения.
- Администратор должен иметь возможность включать experimental lexical features отдельно: query-time синонимы, латиница/кириллица, typo tolerance и trigram fallback. Soundex/rsoundex не входят в базовый контур.
- Trigram fallback должен использовать отдельный индекс, обновляться при записи chunks и иметь admin backfill из уже сохраненных `ai_search_chunks` без embeddings/LLM.
- Trigram backfill должен запускаться как async admin job с endpoint'ами start/status/cancel, прогрессом в UI, structured operational logs и Prometheus metrics.
- `trigramIndexEnabled=true` должен сохраняться только при 100% покрытии trigram index (`chunks > 0`, `trigramChunks >= chunks`, `trigramFtsChunks >= chunks`). Иначе Gateway должен вернуть понятную ошибку готовности.
- Для production-включения trigram нужно пройти staging/readiness benchmark: backfill `completed`, покрытие 100%, нет failed queries, p95 latency trigram stage не выше принятого порога на representative queries, размер SQLite и latency зафиксированы в отчете.
- Qdrant должен вернуть только документы, доступные хотя бы одной группе
  пользователя или публичной группе `*`.
- Gateway должен отбрасывать результаты, доступ к которым нельзя подтвердить.
- Ответ поиска должен включать найденные фрагменты, заголовки страниц и score.

### Чат

- Чат использует тот же ACL-контур, что и поиск.
- Для каждого сообщения контекст из вики подбирается заново.
- История диалога не должна обходить ACL.
- Gateway должен стримить ответ через SSE.
- Ответ должен включать источники, если показ источников включен.
- При недоступности LLM Gateway должен возвращать деградированный ответ с
  найденными источниками, если они есть.

### Индексация

- При изменении страницы MediaWiki должна отправлять событие в Syncer.
- Syncer должен заново получить актуальный контент страницы.
- Старые чанки страницы должны удаляться или заменяться перед записью новых.
- При удалении страницы ее чанки должны удаляться из Qdrant.
- При перемещении или изменении защиты страницы индекс должен обновляться.
- Полная переиндексация должна обходить все страницы MediaWiki.
- После внедрения SMW полная переиндексация должна уметь читать и обновлять
  семантические свойства страниц.
- Индексация SMW-свойств должна иметь dry-run/diagnostic режим без LLM.

### Корпоративный тестовый контент

- Тестовая MediaWiki должна поддерживать повторяемое наполнение через seed-скрипт.
- Минимальный корпоративный набор должен включать 3 департамента, по 3 отдела в
  каждом, 3-4 документа в каждой ветке и отдельный блок общих приказов.
- Набор документов должен покрывать регламенты процессов, пользовательские
  инструкции, инструкции администрирования и FAQ.
- Должны существовать тестовые группы `ai-hr`, `ai-finance`, `ai-it`, `ai-exec`
  и пользователи для проверки разных профилей доступа.
- Для худших ACL-сценариев стенд должен проверять и namespace-level ACL, и
  page-level read ACL через MediaWiki runtime-проверку.
- Seed и live ACL verification не должны выполнять запросы в LLM/OpenAI.

### Вложения

- Файлы MediaWiki рассматриваются как страницы namespace `File:`.
- Для любого файла должны индексироваться минимум метаданные страницы файла.
- PDF и текстовые форматы должны поддерживать извлечение текста.
- Изображения могут обрабатываться через OCR.
- Vision model для схем и диаграмм рассматривается как отключаемый будущий
  уровень обработки.
- ACL вложений должен наследовать модель доступа MediaWiki.

### Администрирование

- `Special:AIAdmin` должен быть доступен только администраторам: `sysop` или
  группе/праву `aiadmin`.
- Админка должна показывать статус Gateway, Qdrant, Redis и LiteLLM.
- Админка должна позволять запускать полную переиндексацию.
- Админка должна показывать прогресс, последние индексированные страницы и логи.
- Админка должна позволять очищать кеши.
- Админка должна показывать runtime-настройки без раскрытия секретов.
- Админка должна позволять управлять политикой распознавания документов по
  MIME-типам: `disabled`, `metadata`, `text`, `ocr`.
- После внедрения SMW админка должна показывать статус SMW-интеграции, количество
  страниц с семантическими свойствами и дату последней semantic reindex операции.
- Админка должна позволять включать AI-разметку только явно и с ограничениями:
  dry-run, лимит страниц, лимит токенов, выбор namespace и запрет автопубликации
  без подтверждения.
- Admin API вызывается из браузера через same-origin `/api/admin/*`, с cookie
  MediaWiki.

## Требования безопасности

- Базовая модель ACL: conservative, deny by default.
- Нельзя показывать документ пользователю, который никогда не имел к нему
  доступа.
- Временный stale access после удаления пользователя из группы считается
  приемлемым до перелогина или истечения кеша.
- Если доступ нельзя подтвердить, документ не должен попадать в результаты.
- Для сложных ACL Gateway должен делать runtime-проверку через MediaWiki API.
- Qdrant и LLM-сервисы не должны иметь публичных endpoint.
- API keys, пароли и shared secrets не должны отображаться в UI.
- Gateway и Syncer не должны напрямую работать с LDAP; они используют группы,
  полученные из MediaWiki.

## Конфигурационные требования

Обязательные внешние параметры:

- `MW_BASE_URL`
- `MW_API_PATH`
- `REDIS_URL`
- `LITELLM_BASE_URL`
- `LITELLM_API_KEY`
- `LITELLM_MODEL`
- `OLLAMA_BASE_URL`
- `OLLAMA_EMBEDDING_MODEL`
- `QDRANT_URL`
- `QDRANT_COLLECTION`

Параметры Gateway:

- `GATEWAY_PORT`
- `NODE_ENV`
- `USER_GROUPS_CACHE_TTL`

Параметры Syncer:

- `SYNCER_PORT`
- `CHUNK_SIZE`
- `CHUNK_OVERLAP`

- `AI_EXTRACTION_MAX_PAGES`
- `AI_EXTRACTION_MAX_TOKENS`

## API-контракты

Минимальные пользовательские endpoint:

- `GET /health` в Gateway.
- `POST /api/search` в Gateway.
- `POST /api/chat` в Gateway.
- `GET /api/admin/health` в Gateway.
- `GET /api/admin/document-processing` в Gateway.
- `POST /api/admin/document-processing` в Gateway.
- `POST /webhook/page` в Syncer.

Ожидаемые административные endpoint должны покрывать:

- получение статуса сервисов;
- чтение и изменение runtime-настроек;
- сброс настроек;
- очистку кеша;
- запуск переиндексации;
- backup/restore или управление snapshot Qdrant.

Будущие endpoint SMW-этапа должны покрывать:

- статус SMW-интеграции;
- dry-run semantic reindex;
- запуск semantic reindex;
- просмотр ontology/properties diagnostics;
- запуск AI extraction только при явном разрешении и лимитах бюджета.

## Нефункциональные требования

- Node.js services должны работать на Node.js 20+.
- TypeScript должен использовать strict mode.
- Gateway должен поддерживать rate limiting.
- Health checks должны проверять критические зависимости.
- Индекс должен переживать перезапуск через volume Qdrant.
- Система должна деградировать безопасно: меньше результатов лучше, чем утечка.
- Документация и конфигурация не должны содержать реальные секреты.

## Ограничения и допущения

- MediaWiki уже развернута и управляет пользователями, сессиями и правами.
- Redis, LiteLLM, Nginx и часть LLM-инфраструктуры могут быть внешними сервисами.
- AD-интеграция является зоной заказчика и выполняется на стороне MediaWiki.
- Namespace ACL является минимальной моделью доступа; page-level ACL требует
  дополнительной интерпретации или runtime-проверки.
- Требования в этом файле описывают целевую модель, а не подтверждают, что каждый
  пункт полностью реализован в текущем коде.
